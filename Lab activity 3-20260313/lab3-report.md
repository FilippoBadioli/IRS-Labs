ALERT_MULTIPLIER ==> Gain applied in moderate obstacle avoidance.
DANGER_MULTIPLIER ==> Gain applied in strong obstacle avoidance.

LEFT_SENSOR, RIGHT_SENSOR, FRONT_SENSOR ==> IDs of the light sensors.
BLACK_THRESHOLD ==> Threshold used to detect the black spot on the ground.
LIGHT_THRESHOLD ==> Threshold to distinguish real light from noise.
MAX_VELOCITY ==> Maximum allowed wheel velocity.

# ARCHITECTURE

The controller is implemented using a **subsumption architecture**, where behaviours are organized in layers of increasing priority.

The architecture is composed of four main behaviours:

* Random Walk (base behaviour)
* Light Following
* Obstacle Avoidance
* Stop (highest priority)

Each behaviour can modify or override the output of the lower-level behaviours. The overall structure is shown in *subsumptionArchitecture.png*.

Additionally, the interaction between behaviours and their internal structure is represented in *moduleArchitecture.png*.
Every behaviour has been designed as a Finite State Machine where, based on the current state, changes its interactions with the lower levels. The only exception is the RandomWalk behaviour due to its simplicity.

# BEHAVIOUR

The robot is able to explore the environment, detect a light source, avoid obstacles, and stop when it reaches the target area.

The behaviour is structured as follows:

* **Random Walk** provides a baseline exploration strategy by generating random wheel velocities.
* **Light Following** is activated when a significant light signal is detected. The robot compares left and right light sensors and adjusts its direction toward the higher intensity.
* **Obstacle Avoidance** has higher priority and modifies the wheel velocities to steer away from nearby obstacles using proximity sensors.
* **Stop** has the highest priority and halts the robot when the black spot under the light source is detected using the ground sensors.

# IMPLEMENTATION

The controller follows a sequential pipeline that emulates the subsumption architecture:

1. A base command is generated by the random walk behaviour.
2. The command is modified by the light-following behaviour if a light source is detected.
3. Obstacle avoidance further adjusts the velocities to ensure safe navigation.
4. The stop behaviour overrides all other commands when the robot is on the target area.

Finally, a velocity scaling mechanism ensures that the wheel speeds do not exceed the maximum allowed value while preserving their relative difference.

This modular structure allows behaviours to be added or modified independently, improving flexibility compared to the finite state machine used in the previous lab.

The controller-subsumption.lua file is organized with blocks of functions:

1. The first block is for the robot base functions init, step, reset and destroy and the controller.
2. The second block are sensor functions which return the values of specific sensors
3. The third block is for function that return the state of each FSM
4. The fourth block is the main body of every behaviour
5. Lastly there is the function used to scale the velocity to not exceed the maximum admitted while still preserving left-right differences

# TESTS EVALUATED AND RESULTS

The controller was tested in different environments with various obstacle configurations, including both boxes and cylinders of different sizes and quantities.

The black spot was positioned below the light source and had a medium size.

The behaviour was also tested with multiple robots (2–3), showing no significant issues in multi-robot scenarios.

The robot is generally able to reach the light source, although convergence can be slow. The main limitations observed are:

* The robot detects the light only when relatively close to it, resulting in prolonged random exploration phases.
* The obstacle avoidance behaviour tends to produce sharp turns instead of smoothly navigating around obstacles.
* The random walk behaviour is highly oscillatory, as direction changes occur at every step.

Despite these limitations, the robot consistently reaches the target area in most scenarios.

# DISCUSSION AND IMPROVEMENTS

Possible improvements include:

* Reducing oscillations in the random walk by updating the turning direction less frequently.
* Refining obstacle avoidance by improving the spatial representation of proximity sensors, allowing smoother navigation around obstacles.
* Increasing sensitivity to light signals to reduce exploration time.

Overall, the subsumption-based architecture provides a more modular and extensible solution compared to the previous FSM approach, allowing behaviours to be layered and combined effectively.
