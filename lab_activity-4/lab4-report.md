# PARAMETERS

The controller relies on a set of parameters that regulate the contribution of each behavioural component:

BASE_VELOCITY ==> Baseline velocity of the robot wheels.  
MAX_VELOCITY ==> Maximum allowed wheel velocity.  
W1 ==> Weight of the attractive potential field generated by the light sensors.  
W2 ==> Weight of the repulsive potential field generated by the proximity sensors.  
W3 ==> Weight of the random exploration component.  

These parameters control the relative influence of each motor schema in the final motion of the robot.

# ARCHITECTURE

The controller is implemented using a **motor schema approach combined with potential fields**.

Unlike the subsumption architecture used in the previous lab, behaviours are not organized in layers with priorities, but instead contribute simultaneously to the final motion through vector combination.

The architecture is composed of three main components:

* Light attraction field  
* Obstacle repulsion field  
* Random exploration field  

Each component generates a vector in polar coordinates (length and angle). The overall behaviour emerges from the **vectorial sum of these fields**.

This approach results in a continuous control strategy, where all behaviours are active at the same time and their influence is modulated through weights.

# BEHAVIOUR

The robot is able to explore the environment, approach a light source, and avoid obstacles through the interaction of multiple motor schemas.

The behaviour is structured as follows:

* **Light Attraction** generates a vector pointing toward the direction of higher light intensity. The contribution of each light sensor is combined to form a global attractive field.
* **Obstacle Avoidance** generates a repulsive vector by reversing the direction of the proximity sensor readings, pushing the robot away from obstacles.
* **Random Exploration** provides a constant forward component that ensures the robot keeps moving even in absence of strong stimuli. For our assumption about the arena configuration this is enough to ensure a decent random exploration, but would require a more complex implementation in case those assumptions are no longer true.

The final motion results from the combination of these components, allowing the robot to smoothly navigate toward the goal while avoiding collisions.

# IMPLEMENTATION

The controller follows a perception → vector computation → combination → actuation pipeline:

1. Sensor readings from light and proximity sensors are converted into vectors using polar coordinates.
2. Each vector field is computed independently.
3. The resulting vectors are combined using vector summation.
4. The final vector is converted into linear and angular velocities.
5. Wheel velocities are computed using differential drive kinematics.
6. A scaling function ensures that the maximum velocity constraint is respected while preserving the relative difference between wheels.

The controller-potential_fields.lua file is organized as follows:

1. Initialization and main control loop (init, step, controller).
2. Functions for computing individual vector fields.
3. Function for converting polar vectors into wheel velocities.
4. Velocity limiting function.

# TESTS EVALUATED AND RESULTS

The controller was tested in different environments with various obstacle configurations, including boxes and cylinders of different sizes and quantities.

The robot is generally able to reach the light source while avoiding obstacles. Compared to the subsumption-based controller, the motion appears smoother and more continuous.

However, some limitations were observed:

* The robot may get temporarily stuck in local configurations where attractive and repulsive forces balance each other.
* Convergence speed depends heavily on the tuning of the weights.

Despite these limitations, the robot consistently demonstrates the ability to reach the target in most scenarios.

# DISCUSSION AND IMPROVEMENTS

Possible improvements include:

* Introducing adaptive weights to reduce the influence of the random component when a strong light signal is detected.
* Refining the obstacle field by considering spatial grouping of proximity sensors to improve navigation around obstacles.
* Introducing a more structured random exploration behaviour to find the light source faster.
* Addressing local minima situations by adding noise or alternative escape strategies.

Overall, the motor schema approach provides a smooth and reactive behaviour, allowing continuous interaction between different control components and improving motion quality compared to discrete architectures.
